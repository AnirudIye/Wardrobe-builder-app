# Deploying BetterDresser

The codebase is deploy-ready and platform-agnostic. This doc gives one concrete
path (Neon + Cloudflare R2 + Render + Vercel, all free-tier) plus the generic
checklist. Two config files do most of the work:

- **`render.yaml`** (repo root) - Render Blueprint for the FastAPI backend.
- **`frontend/vercel.json`** - Vercel config for the SPA: `/api/*` proxy to the
  backend, security headers, CSP with `frame-ancestors`.

Recommended order: **database → media bucket → backend → frontend → wire the
cross-references (CORS, OAuth origin, Stripe webhook).**

---

## 0. Python version

Production runs **Python 3.12** (pinned in `render.yaml` via `PYTHON_VERSION`
and in `backend/.python-version`). The full test suite passes on 3.12.13; the
only warning is an upstream `python-jose` `datetime.utcnow()` deprecation, which
is benign. The local dev `.venv` can stay on whatever it is - the pin only
affects deploy targets and pyenv.

## 1. Database - Neon (managed Postgres)

1. Create a project at [neon.tech](https://neon.tech) (free tier: 0.5 GB,
   scale-to-zero).
2. Copy the connection string and convert it to the psycopg3 form:
   `postgresql+psycopg://USER:PASSWORD@HOST/DBNAME?sslmode=require`
3. That string is your `DATABASE_URL`. The schema is applied by
   `alembic upgrade head`, which `render.yaml` runs automatically on every boot
   (see section 3) - you don't run it by hand.

The engine already sets `pool_pre_ping` for Neon's autosuspend. Supabase
Postgres also works, but note the pooler caveat in `backend/.env.example`.

## 2. Media - Cloudflare R2 (or any S3-compatible store)

Render's free disk is **ephemeral** - uploaded photos vanish on redeploy - so
media must live in object storage. The S3 backend is already coded.

1. Create an R2 bucket at Cloudflare, enable public access (or a public dev
   URL / custom domain).
2. Create an R2 API token (Access Key ID + Secret).
3. You'll set these on the backend (section 3): `STORAGE_BACKEND=s3`,
   `S3_BUCKET`, `S3_ENDPOINT_URL` (`https://<account_id>.r2.cloudflarestorage.com`),
   `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY`, `S3_PUBLIC_BASE_URL` (the bucket's
   public URL). `S3_REGION` defaults to `auto`, which is correct for R2.

Startup **fails fast** if `STORAGE_BACKEND=s3` without `S3_BUCKET` +
`S3_PUBLIC_BASE_URL`, so you can't half-configure it by accident.

## 3. Backend - Render (Blueprint)

1. Push this repo to GitHub.
2. Render dashboard → **New + → Blueprint** → pick the repo. Render reads
   `render.yaml` and creates the `betterdresser-api` web service (Python 3.12,
   `rootDir: backend`).
3. In the service's **Environment** tab, fill in every `sync: false` var
   (Render leaves them blank on purpose - never commit secrets):
   - `DATABASE_URL` (section 1)
   - the six `S3_*` vars (section 2)
   - `CORS_ORIGINS` and `FRONTEND_BASE_URL` - **set after Vercel gives you a URL**
     (section 4); both are your frontend origin, e.g.
     `https://betterdresser.vercel.app`
   - feature keys you want live: `ANTHROPIC_API_KEY` and/or `GOOGLE_API_KEY`
     (text AI + TryOn), `GOOGLE_OAUTH_CLIENT_ID` (Google sign-in),
     `OPENWEATHER_API_KEY`, `SERPAPI_KEY`, `STRIPE_*`, `SMTP_*`. Anything left
     unset degrades gracefully - the app still boots and runs.
   - `JWT_SECRET` is generated by Render automatically; `ENVIRONMENT=production`
     and `PYTHON_VERSION` are set by the blueprint.
4. The start command is `alembic upgrade head && uvicorn app.main:app … --proxy-headers`.
   Migrating in the start command keeps the schema step explicit and works on
   the free plan. **On a paid plan**, move it to `preDeployCommand:` in
   `render.yaml` so migrations run once before the new version goes live rather
   than on every boot.
5. Note the service's URL (e.g. `https://betterdresser-api.onrender.com`).
   `render.yaml` names the service `betterdresser-api`, so that's usually the
   default URL - but confirm it and use it in the next step.

Render terminates TLS and sets `ENVIRONMENT=production`, so the API sends HSTS,
disables `/docs`, and rate-limits auth per real client IP (via `--proxy-headers`).

## 4. Frontend - Vercel

1. Vercel dashboard → **Add New → Project** → pick the repo.
2. Set **Root Directory = `frontend`**. Vercel then reads `frontend/vercel.json`
   (framework `vite`, build `npm run build`, output `dist`).
3. **Point the proxy at your backend.** `vercel.json` rewrites `/api/*` to
   `https://betterdresser-api.onrender.com/*` by default. If your Render URL is
   different, edit the `rewrites[0].destination` in `frontend/vercel.json`. The
   rewrite strips `/api` (matching the dev Vite proxy), and because the browser
   only ever talks to the Vercel origin, there's **no CORS and no exposed API
   domain** - `connect-src 'self'` in the CSP just works.
4. Deploy. Vercel gives you a URL (e.g. `https://betterdresser.vercel.app`).
5. **Back-fill the cross-references** now that both URLs exist:
   - On Render: set `CORS_ORIGINS` and `FRONTEND_BASE_URL` to the Vercel URL,
     then redeploy the backend. (The API is reached server-side through the
     proxy, so CORS isn't browser-enforced here - but production startup
     *requires* `CORS_ORIGINS`, and it's correct if anything ever calls the API
     directly.)
   - Google sign-in: add the Vercel URL to **Authorized JavaScript origins** on
     the OAuth client (console.cloud.google.com), and publish the consent screen.
   - Stripe: set the webhook endpoint to `https://<vercel-url>/api/billing/webhook`
     and put the signing secret in `STRIPE_WEBHOOK_SECRET` on Render.

`vercel.json` also serves the CSP as a real response header (with
`frame-ancestors 'none'`, which the build-time `<meta>` tag can't express) plus
`nosniff` / `Referrer-Policy` / `X-Frame-Options` / `Permissions-Policy`
(`camera=(self)` for TryOn) / HSTS - matching the API's own headers.

### Upload size note

Wardrobe/TryOn photos are proxied through Vercel's rewrite. If very large
uploads ever fail at the edge, the escape hatch is to call the Render API on its
own subdomain directly (add a `VITE_API_BASE` to `frontend/src/api.ts`, enable
CORS via `CORS_ORIGINS`, and add that origin to the CSP `connect-src`). Not
needed for typical phone photos; documented in case.

## Generic checklist (any host)

| Key | Production value |
| --- | --- |
| `ENVIRONMENT` | `production` (startup fails fast on weak config) |
| `JWT_SECRET` | `python -c "import secrets; print(secrets.token_urlsafe(32))"` |
| `JWT_EXPIRE_MINUTES` | `4320` (3 days) |
| `CORS_ORIGINS` | exact frontend origin(s), e.g. `https://app.example.com` |
| `DATABASE_URL` | managed Postgres (`postgresql+psycopg://...?sslmode=require`) |
| `STORAGE_BACKEND` + `S3_*` | R2/S3 bucket (free-tier host disks are ephemeral) |
| `FRONTEND_BASE_URL` | the deployed frontend URL (Stripe redirects, verify/reset links) |

- Terminate TLS at the platform; do NOT add an app-level HTTPS redirect.
- Run uvicorn with `--proxy-headers --forwarded-allow-ips='*'` so rate limiting
  sees real client IPs.
- Rotate any secret ever pasted into a chat, commit, or ticket (the Gmail app
  password, the NVIDIA key, and the old Brevo keys all qualify).

## Already handled in code (don't undo it)

- Security headers on every API response; `/docs` disabled in prod.
- Auth endpoints rate-limited + per-account lockout (in-process - swap to Redis
  before trusting exact limits past one worker).
- Remote image fetches SSRF-guarded (public-IP-only, per redirect hop).
- Stripe webhooks signature-verified; media keys are unguessable uuid4.
- Password hashing (bcrypt), email verification, JWT expiry, self-serve data
  export + account deletion (GDPR).
